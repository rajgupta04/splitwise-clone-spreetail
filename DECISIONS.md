# Engineering Decisions Log

Tracks key architectural and technical design decisions, alternatives considered, pros/cons, and final choice reasonings.

---

## DEC-001: ORM and Database Access Layer

* **Decision**: Choose the database access layer strategy for handling relational data and temporal queries.
* **Alternatives Considered**: 
  1. **Prisma ORM only**: Use Prisma's standard API for all queries.
  2. **Raw SQL only**: Use raw queries (via `pg-pool`) for all operations.
  3. **Hybrid approach**: Use Prisma for standard CRUD operations and schema management, and raw SQL (`$queryRaw`) only for complex temporal queries.
* **Pros**:
  * *Prisma only*: High developer ergonomics, auto-generated TypeScript/JS types, safe database migrations.
  * *Raw SQL only*: Maximum performance, full control over query execution plans, native support for complex joins and date comparisons.
  * *Hybrid*: Combines the speed and convenience of Prisma CRUD with the flexibility of raw SQL for complex queries.
* **Cons**:
  * *Prisma only*: Inability to express complex date-range intersections elegantly (e.g. `joined_at <= date AND (left_at IS NULL OR left_at >= date)`).
  * *Raw SQL only*: Wasted developer overhead writing basic select/insert/update boilerplate; migrations must be managed manually.
  * *Hybrid*: Introduces two database access patterns, increasing maintenance overhead.
* **Final Choice**: Hybrid Approach (Prisma ORM + raw SQL `$queryRaw` when needed).
* **Reasoning**: Prisma client handles 95% of standard CRUD operations elegantly (e.g. user authentication, expense creation, CSV uploads). However, temporal queries checking user memberships on specific transaction dates require date-range comparisons that cannot be expressed nicely in Prisma's query syntax. Reserved raw SQL specifically for those edge cases.

## DEC-002: Primary and Foreign Key Column Types

* **Decision**: Select the column data types to use for IDs and relationships.
* **Alternatives Considered**:
  1. **PostgreSQL Native UUID**: `@db.Uuid` annotations in the schema.
  2. **PostgreSQL TEXT**: Default Prisma `String` fields (without annotations).
* **Pros**:
  * *UUID*: Highly performant database indexing, native UUID validation at database engine level.
  * *TEXT*: Database-agnostic schema design, default mapping in Prisma without custom annotations.
* **Cons**:
  * *UUID*: Requires database-specific types, database migrations are more complex.
  * *TEXT*: Requires explicit casts (`::text`) in raw queries to bypass Prisma's client-level auto-inference of UUID-formatted string parameters.
* **Final Choice**: TEXT Column Type (Prisma default `String`).
* **Reasoning**: The schema was initialized using Prisma's default `String` representation for identifiers. It uses UUID-formatted strings generated by Javascript/Prisma. Although it requires defensive casting in raw SQL, it maps cleanly to Prisma defaults.

## DEC-003: Multi-Currency Support Strategy

* **Decision**: Design the currency modeling and storage approach.
* **Alternatives Considered**:
  1. **Normalize on write**: Convert all amounts to the group's base currency using a live API at write time and discard original values.
  2. **Audit-safe dual-amount storage**: Store the original transaction amount, original currency, conversion rate used, and normalized base currency amount.
* **Pros**:
  * *Normalize on write*: Simple database structure; no currency columns needed on splits.
  * *Dual-amount storage*: Maintains full audit reproducibility; historical records do not lose context if group currencies change or if conversions need audit.
* **Cons**:
  * *Normalize on write*: Destructive data loss (original payment amount is lost); dependent on live API rates (making imports non-reproducible).
  * *Dual-amount storage*: Additional table columns, increased math complexity on splits.
* **Final Choice**: Audit-safe dual-amount storage.
* **Reasoning**: Real-world expenses need auditability. Storing the original values, exchange rates, and normalized amounts ensures that historical calculations remain reproducible without dependency on external rate changes.

## DEC-004: Temporal Membership Implementation

* **Decision**: Implement a system that allows users to join and leave groups without corrupting historical balances.
* **Alternatives Considered**:
  1. **Flat membership list**: Simple join table with no timestamps.
  2. **Temporal audit log**: Maintain a timeline of `joined_at` and `left_at` timestamps for every member.
* **Pros**:
  * *Flat*: Extremely simple queries (user is either a member or not).
  * *Temporal*: Enables users to join/leave/rejoin dynamically; ensures that expenses logged in February do not affect members who joined in April.
* **Cons**:
  * *Flat*: Destructive exit (leaving wips historical tracking), or incorrect balance impact (new members are charged for old bills).
  * *Temporal*: Requires complex raw SQL checks on every expense and balance calculation to determine who was active.
* **Final Choice**: Temporal audit log memberships (`joined_at`, `left_at`, `status`).
* **Reasoning**: A flat list causes severe business logic bugs (e.g. new users inherit past debts, or leaving a group erases a user's unpaid balances). Temporal timelines ensure exact accounting integrity.

## DEC-005: Currency Service Caching

* **Decision**: Choose the caching strategy for the Frankfurter API exchange rates.
* **Alternatives Considered**:
  1. **Persistent Cache (Redis/PostgreSQL)**: Store fetched exchange rates in a database table or Redis cache.
  2. **In-Memory JavaScript Map**: Store fetched rates in memory during the application lifecycle.
* **Pros**:
  * *Persistent Cache*: Survives server restarts; shares cache across multiple application instances.
  * *In-Memory Map*: Zero infrastructure overhead; extremely fast reads; no database schema modifications needed.
* **Cons**:
  * *Persistent Cache*: Requires database changes or new infrastructure (Redis).
  * *In-Memory Map*: Cache is cleared on server restart; multiple server instances would fetch duplicate data initially.
* **Final Choice**: In-Memory JavaScript Map.
* **Reasoning**: The application currently runs as a single instance and does not use Redis. Since historical exchange rates never change, an in-memory map keyed by `date_base_target` provides a simple, highly effective cache without adding infrastructure complexity. This satisfies the requirement for rate caching during batch operations like CSV imports.

## DEC-006: Group Base Currency Strategy

* **Decision**: Establish a strict base currency for each group, and fetch real-time or historical rates when a foreign currency is used.
* **Alternatives Considered**:
  1. **Floating User Preferences**: Each user sees balances in their own preferred currency, calculated on the fly.
  2. **Strict Group Base Currency**: A group has exactly one base currency, and all cross-currency transactions are immediately normalized to this base at the time of entry.
* **Pros**:
  * *Floating*: High personalization for international users.
  * *Strict Base Currency*: Ensures deterministic, auditable, and reproducible balances that never change retroactively due to foreign exchange market fluctuations.
* **Cons**:
  * *Floating*: Balances fluctuate daily; creates immense confusion when reconciling debts ("You owe me $10 today, but $11 tomorrow").
  * *Strict Base Currency*: Requires tracking explicit exchange rates at the moment an expense is added.
* **Final Choice**: Strict Group Base Currency.
* **Reasoning**: Balances must be reproducible, auditable, and independent of future exchange-rate changes. By fetching the exchange rate exactly once at creation/import and storing the `normalized_amount` along with the `exchange_rate` used, we guarantee financial stability within the group.
